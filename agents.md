# 项目开发规则

1. 单个文件不能超过 1000 行。

2. 修改了对应的页面后，要在对应页面右上方增加一个“业务及需求说明”以及修改时间。

3. “业务及需求说明”主要讲解该模块的功能及作用，以及关联模块。

4. 当前所有页面后续修改时，都需要在页面右上角新增“业务及需求说明”入口。

5. 不用每次修改都上传到 Git；我会在需要时明确下达上传到 Git 的指令。

6. 总控后台左侧菜单栏需要在“首页”上方保留“本周需求说明”入口；进入后展示周说明区，每周新增一列，列内按“总控后台 / 站点后台 / 代理后台”展示本周最新需求内容、模块说明、修改说明和页面跳转。

7. 每次完成页面或功能修改后，需要同步更新总控后台“本周需求说明”页面对应周的内容；同一周内反复修改时，不追加重复历史记录，用最新业务说明覆盖该周原说明。

8. 每次修改页面或功能后，需要同步更新该页面右上角“业务及需求说明”的最新内容，说明必须包含页面功能说明、字段说明、业务逻辑说明、关联模块、业务及需求说明和功能验收说明，文案面向产品与运营。

9. “本周需求说明”中的新增或修改模块，必须同步更新该模块详情页右上角“业务及需求说明”；模块详情页的业务及需求说明要写明完成时间、修改内容、业务作用、关联模块和功能验收说明。

10. “本周需求说明”必须按“总控后台 / 站点后台 / 代理后台”分组展示；每个新增或修改模块必须写明完成时间、模块说明、修改说明和页面跳转；非业务需求不写入本周需求说明。

11. 同一模块在同一周内反复调整时，不堆叠重复历史记录，用最新业务说明覆盖；文案不得出现代码、接口、数据库等技术内容。

12. 未明确要求同步 Git 时，只做本地修改和验证，不提交、不推送。

13. GitHub Pages 发布仓库推送地址为 `git@github.com:npc-sikong/master-admin-prototype-demo.git`；如本机直连 GitHub SSH 22 端口超时，可使用 GitHub SSH 443 通道推送。

14. 业务逻辑说明只针对新增或修改过的模块做增强；旧模块不批量扩写。增强时优先围绕本次修改内容说明查询/查看逻辑、操作逻辑、状态逻辑、金额或次数逻辑、关联模块和演示边界。

15. “业务及需求说明”仍为演示原型内置文案，不做页面内编辑保存功能；涉及金额或次数的业务逻辑说明必须补充产品和运营可理解的计算公式。

16. 每次新增或修改模块后，该模块右上角“业务及需求说明”必须补充“修改记录”，每条记录需包含修改时间、修改说明和修改内容；修改说明讲清业务目的，修改内容列出页面、筛选、字段、弹窗、规则或跳转等实际变化。同一业务项反复微调时可覆盖为最新说明，不写过程流水账。
